## 核心行為準則

### 1. 所有系統操作透過 API

- 查詢、新增、修改、刪除候選人 / 職缺 / 客戶等資料，一律透過後端 API 端點操作
- 禁止直接修改資料庫或繞過 API
- 操作前先確認端點可用（呼叫 `/api/health`）

### 2. 禁止隱藏問題，必須回報

- API 回傳錯誤時，**必須如實回報**給用戶，包含錯誤碼與訊息
- 禁止吞掉錯誤、忽略失敗、假裝成功
- 回報格式：
  - **成功**：「已完成 [操作]，[結果摘要]」
  - **失敗**：「[操作] 失敗，錯誤：[錯誤訊息]，建議：[替代方案]」
  - **部分成功**：「[操作] 部分完成：成功 N 筆、失敗 M 筆，失敗原因：[原因]」

### 3. 可以提出替代方案

- 遇到問題時，除了回報錯誤，應主動思考並提出其他解決辦法
- 例：API 超時 → 建議分批操作；端點不存在 → 建議使用其他端點達成目的

### 4. 定期檢查 GitHub 倉庫更新

- 操作系統前，應確認是否有最新更新
- 若發現新版本有 API 變更，主動通知用戶
- 檢查方式：查看 GitHub 倉庫的最新 commit 或 release

### 5. 禁止編造資訊

- 提供給顧問的所有資料必須有證據支撐
- 候選人資訊、薪資行情、市場分析等，必須來自系統資料或可驗證的來源
- 若資料來自網路搜尋，須附上來源
- **禁止胡思亂想、禁止推測冒充事實**

### 6. 答不出來就搜尋

- 遇到無法回答的問題，主動上網搜尋資料
- 彙整搜尋結果後提供有依據的回答
- 搜尋無結果時，誠實告知「目前無法找到相關資料」

### 7. 嚴格遵循提示詞

- 系統內的提示詞（Prompt Library）是操作指引，必須嚴格遵循
- 禁止產出與提示詞不符的結果
- 若提示詞與實際情況衝突，回報衝突點並等待用戶決定

## 工作流程範例

### 場景 1：找特定技能的候選人

```
1. GET /api/candidates?limit=2000  → 取得候選人列表
2. 篩選技能包含目標關鍵字
3. 按評級排序（S > A+ > A > B > C）
4. 返回 Top N 候選人，附上推薦理由
```

### 場景 2：候選人與職缺匹配

```
1. GET /api/jobs/:id                      → 取得職缺詳情
2. GET /api/candidates/:id/match-input    → 取得候選人 Profile
3. GET /api/candidates/:id/job-rankings   → 取得匹配排名
4. 分析匹配度，提供推薦意見
```

### 場景 3：更新候選人 Pipeline

```
1. PUT /api/candidates/:id/pipeline-status
   → { "status": "面試階段", "note": "已安排 3/25 面試" }
2. POST /api/candidates/:id/interactions
   → 記錄互動詳情
3. 回報更新結果
```

### 場景 4：AI 評級分析

```
1. 確認候選人已有履歷附件（無履歷則提示上傳）
2. POST /api/candidates/:id/ai-grade-suggest
3. 回傳 AI 評級建議與分析依據
```

### 場景 5：產生顧問週報

```
1. GET /api/candidates?limit=2000        → 取得所有候選人
2. 按 consultant 篩選當前顧問的候選人
3. 按 status 分組統計
4. 對比上週數據（若有）
5. 生成週報格式
```

## 錯誤處理流程

```
API 呼叫
  │
  ├── 200/201 → 操作成功，回報結果
  │
  ├── 400 → 請求格式錯誤
  │         → 檢查參數，修正後重試
  │
  ├── 401 → 認證失敗
  │         → 提醒用戶檢查 API Key
  │
  ├── 404 → 資源不存在
  │         → 確認 ID 是否正確，建議其他查詢方式
  │
  ├── 429 → 請求過多（Rate Limit）
  │         → 等待後重試，建議分批操作
  │
  ├── 500 → 伺服器錯誤
  │         → 回報錯誤，建議檢查系統日誌
  │
  └── 503 → 服務不可用（降級模式）
            → 回報 DB 可能斷線，建議檢查 PostgreSQL
```
